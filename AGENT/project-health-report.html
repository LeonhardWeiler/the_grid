1 · Offene Findings

kritisch Grid-Größe zwischen Frontend und Backend inkonsistent CRIT-1

frontend/src/camera/camera.ts:3 setzt GRID_SIZE = 1000, das Backend internal/config.go:7 jedoch GridSize = 512.

Das Frontend zeigt und erlaubt Klicks auf ein 1000×1000-Raster. Klicks außerhalb 0–511 werden per set_pixel gesendet, vom Backend in ValidCoords (internal/validate.go:9) still verworfen — kein Fehler, kein Feedback, kein Cooldown. Über die Hälfte der sichtbaren Fläche ist unbrauchbar. Zusätzlich rechnet toCenteredCoords mit Mittelpunkt 500 statt 256, wodurch die angezeigten Koordinaten nicht zum echten Raster passen.

Empfehlung: Eine einzige Quelle der Wahrheit für die Grid-Größe. Backend soll die Grid-Größe beim init an den Client senden, das Frontend nutzt diesen Wert statt einer eigenen Konstante.

kritisch Nach Reload bleibt die Canvas nahezu leer (Delta-Sync ohne Pixel-Persistenz) CRIT-2

frontend/src/ws.ts:27,80,118 persistiert nur version in localStorage, nicht die Pixel-Daten. Beim Reload ist die pixels-Map leer, aber es wird init_client mit der alten, hohen lastVersion gesendet.

Das Backend (internal/ws.go:108) liefert dann nur Deltas seit dieser Version (sync), sofern der Ring-Buffer sie noch abdeckt. Der Client wendet diese wenigen Events auf eine leere Map an → es erscheinen nur die seit dem letzten Besuch geänderten Pixel, der Rest der Canvas fehlt, bis er zufällig überschrieben wird.

Empfehlung: Entweder beim Reconnect/Reload immer einen vollständigen Snapshot anfordern (lastVersion = 0, wenn keine lokalen Pixel vorliegen), oder die Pixel-Map ebenfalls lokal persistieren. Delta-Sync nur, wenn tatsächlich ein konsistenter lokaler Zustand existiert.

hoch docker-compose Volume zeigt auf falschen Pfad — Persistenz geht verloren HIGH-1

docker-compose.yml:9 mountet ./data:/data. Der Server schreibt aber nach ./data/snapshot.json relativ zum WORKDIR /app (internal/hub.go:36, Dockerfile:21), also nach /app/data.

Der gemountete Volume-Pfad /data wird nie beschrieben; die Snapshots landen im ephemeren Container-Layer und gehen bei jedem docker compose up --force-recreate verloren.

Empfehlung: Volume auf ./data:/app/data ändern (oder den Snapshot-Pfad im Server auf /data konfigurierbar machen und dort hin schreiben).

hoch WebSocket-Origin-Whitelist nur für localhost — Produktion blockiert HIGH-2

internal/ws.go:55 erlaubt als OriginPatterns ausschließlich http://localhost:5173 und http://localhost:4000.

Beim Deployment (Docker, Port 4000) wird die App über eine Domain/IP aufgerufen; die Origin lautet dann z. B. http://<host>:4000 und ist nicht in der Liste → websocket.Accept lehnt mit 403 ab. Alle WS-Verbindungen scheitern, die App bleibt im „Connecting…"-Zustand hängen.

Empfehlung: Erlaubte Origins über eine Env-Variable konfigurierbar machen und in der Deploy-Konfiguration den echten Host eintragen.

hoch Cooldown per selbstgewählter clientId trivial umgehbar HIGH-3

Das Rate-Limiting (internal/hub.go:178) verwendet die vom Client gelieferte clientId als Schlüssel (internal/ws.go:98). Ein Client kann für jeden Pixel eine neue zufällige, regex-konforme ID senden und so den 5-Sekunden-Cooldown komplett aushebeln.

Empfehlung: Cooldown zusätzlich (oder primär) an eine serverseitige Größe binden, z. B. Remote-IP der Verbindung. Client-ID bleibt für Session-Zuordnung, ist aber nicht allein rate-limit-tragend.

mittel lastAction-Map wächst unbegrenzt MED-1

internal/hub.go:239 definiert CleanupLastAction, aber die Funktion wird nirgends aufgerufen. Für jede jemals gesehene clientId bleibt ein Eintrag dauerhaft im Speicher — ein langsames, unbegrenztes Leck (in Verbindung mit HIGH-3 sogar beschleunigt).

Empfehlung: Cleanup periodisch per Ticker (analog zu StartAutosave) mit einer Max-Alters-Schwelle aufrufen.

mittel Kein WebSocket-Heartbeat / kein Read-Deadline MED-2

internal/ws.go:78 liest mit einem nie ablaufenden Context; es gibt kein Ping/Pong und keinen Read-Timeout. Tote Verbindungen (Netzwerkabbruch ohne Close-Frame) bleiben samt Goroutine und Map-Einträgen hängen.

Empfehlung: Periodische Pings mit Pong-Deadline einführen oder pro Read einen Timeout-Context setzen; bei Ausbleiben Verbindung schließen. In README/CLAUDE.md bereits als geplant vermerkt.

mittel Docker-Frontend-Build ignoriert vite.config.ts und hat kein bun-Lockfile MED-3

Dockerfile:1-9 kopiert nur frontend/ und die Root-package.json, aber nicht die vite.config.ts (liegt im Repo-Root). Der Build läuft daher mit Vite-Defaults statt mit root:"frontend"/outDir:"../public/dist" — er funktioniert nur zufällig, weil die Defaults zum Copy-Layout passen. Zudem verweist COPY … bun.lockb* auf eine nicht vorhandene Lockdatei (Repo nutzt package-lock.json), sodass bun install ohne Lock nicht reproduzierbar auflöst.

Empfehlung: Build-Kontext und Vite-Config konsistent halten (Config mitkopieren oder Frontend-Build vollständig in frontend/ kapseln) und ein echtes Lockfile committen/kopieren.

mittel Keine Touch-Bedienung & fehlendes Viewport-Meta — auf Mobil unbenutzbar MED-4

frontend/src/camera/controls.ts registriert ausschließlich mousedown/mousemove/wheel — kein touchstart/touchmove und kein Pinch-Zoom. frontend/index.html hat kein viewport-Meta-Tag. Auf Touch-Geräten lässt sich weder pannen noch zoomen noch zuverlässig ein Pixel setzen.

Empfehlung: Touch-/Pointer-Events für Pan & Pinch-Zoom ergänzen und ein <meta name="viewport"> setzen. In README als „Mobile touch controls" bereits als Idee gelistet.

mittel Kein Nutzer-Feedback bei abgelehntem Pixel MED-5

Wird ein set_pixel serverseitig verworfen (out of bounds, ungültige Farbe, Cooldown aktiv), erfolgt keinerlei Antwort. Das Frontend setzt selectedPixel optimistisch zurück (frontend/src/main.ts:67) und der Nutzer erhält keine Rückmeldung, warum nichts passiert ist.

Empfehlung: Eine kompakte rejected/error-Nachricht vom Server senden und im UI kurz anzeigen (z. B. Toast oder rot blinkender Cooldown-Button).

niedrig MaxMessageSize ungenutzt — kein Read-Limit gesetzt LOW-1

internal/config.go:9 definiert MaxMessageSize = 1024, aber die Konstante wird nie verwendet; auf der Verbindung ist kein SetReadLimit gesetzt, es gilt still das Bibliotheks-Default (32 KiB).

Empfehlung: conn.SetReadLimit(MaxMessageSize) nach dem Accept setzen oder die tote Konstante entfernen.

niedrig Toter Code & doppelte Verzeichnis-Anlage LOW-2

internal/hub.go:252 StartPersistenceLoop wird nie aufgerufen (Autosave läuft über autosave.go). Zusätzlich legen sowohl NewHub (internal/hub.go:31) als auch NewPersistence (internal/persistence.go:20) das data-Verzeichnis an — mit uneinheitlichem Pfad (./data vs. data).

Empfehlung: Ungenutzte Funktion entfernen und die Verzeichnis-Anlage an einer Stelle bündeln.

niedrig Accessibility der Palette und Canvas LOW-3

frontend/index.html:15-54: die Farb-Buttons haben keinen zugänglichen Namen (nur Hintergrundfarbe), die Canvas ist nicht per Tastatur bedienbar und die aktive Farbe wird nur visuell (Klasse active) markiert.

Empfehlung: aria-label mit Farbnamen, aria-pressed für die aktive Farbe und optionale Tastatur-Navigation für die Pixel-Auswahl ergänzen.

Im gesamten Repo existieren weder Go- noch Frontend-Tests. Kritische Kernlogik (Ring-Buffer in store.go, GetSince-Grenzfälle, Cooldown, Koordinaten-Umrechnung) ist ungeschützt gegen Regressionen — gerade die Sync-Grenzfälle aus CRIT-2 wären hier auffällig geworden.

Empfehlung: Mit gezielten Unit-Tests für PixelStore (Ring-Wraparound, GetSince unter/über dem Fenster) und die Koordinaten-Utils starten; im CI ein go test ./...-Gate ergänzen.

2 · Was schon stark ist

  • Atomare Snapshot-Writes: persistence.go schreibt via tmp → rename, kein Halb-Zustand bei Crash.
  • Sauberes Shutdown & Autosave: autosave.go deckt periodisches Speichern und SIGINT/SIGTERM-Save ab.
  • Server-Autorität konsequent: alle Pixel werden serverseitig validiert (ValidCoords/ValidColor/ValidClientID), Clients senden nur Intents.
  • Effizienter Ring-Buffer mit fester Kapazität verhindert unbegrenztes Event-Wachstum.
  • Robustes Reconnect: exponentielles Backoff + Jitter + Online/Offline-Handling im WS-Client.
  • Saubere Nebenläufigkeit: durchgängige RWMutex-Nutzung im Hub, per-Client Send-Channel mit Drop bei Überlauf.
  • Grüne Gates: go vet, go build und tsc --noEmit laufen fehlerfrei; TS-Config ist strikt (strict, noUncheckedIndexedAccess).

3 · Bewusst entschieden

  • In-Memory-Store + JSON-Snapshot statt Datenbank — passt zum Ziel „hackbar & einfach", bewusst nicht auf Skalierung optimiert.
  • Feste 6-Farben-Palette (config.go / index.html) — minimalistisch im r/place-Geist.
  • Text-basiertes JSON-WS-Protokoll statt Binärprotokoll — Lesbarkeit über Performance (Binär ist als Zukunft notiert).
  • Kein Frontend-Framework — direkter Canvas + TypeScript als ausdrückliches Designprinzip.
